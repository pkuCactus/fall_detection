# 纯视觉端侧跌倒检测系统设计

**日期**: 2026-03-31  
**目标平台**: 海思 Hi3516CV500 / 3516C（0.5T INT8，30M Flash，15M DDR）  
**输入**: 1920×1080 @ 25fps  

---

## 1. 设计目标与核心约束

### 1.1 设计目标
构建一套可在安防芯片端侧部署的纯视觉跌倒检测系统，方案为：

> 人体检测 + 多目标跟踪 + 人体关键点估计 + 姿态分类 + 规则/融合决策

### 1.2 硬件约束
- **算力**: 0.5T INT8
- **Flash**: 可用 30M
- **DDR**: 可用 15M
- **推理框架**: 海思 NNIE/WK 工具链（训练侧以 PyTorch → ONNX → 海思格式为部署主线）

这些约束要求模型总权重控制在 **6M 以内**，后处理必须极度精简。

---

## 2. 整体架构

系统由 6 个模块级联组成：

```
输入 1920×1080 @ 25fps
   ↓（抽帧：每 2~3 帧跑一次检测）
[模块1] 人体检测器（YOLOv8n，832×448 INT8）
   ↓ 检测框 + 置信度
[模块2] ByteTrack-lite（卡尔曼 + 匈牙利，无 ReID 网络）
   ↓ Track ID + ROI
[模块3] 17 关键点姿态估计（YOLOv8n-pose，全帧推理）
   ↓ 17 keypoints + visibility
[模块4] 分类器（单分支图像 / 融合 3 分支）
   ↓ 跌倒概率 S_cls（每帧都推理；跳帧时复用缓存关键点）
[模块5] 规则引擎（A/B/C/D/E/F 六条规则，接收 S_cls 辅助姿态分类）
   ↓ 疑似跌倒得分 S_rule + 修正后姿态
[模块6] 融合决策器（规则 + 分类器 + 时序投票 + 状态机 → 最终事件输出）
   ↓
输出：报警事件 + 目标框 + 关键点 + 跌倒置信度
```

### 2.1 模块1：人体检测器
- **选型**: NanoDet-plus-m-0.5x 或 PP-YOLOE-s with MobileNetV2 backbone 的裁剪版
- **输入尺寸**: `832×448`（保持较大分辨率以适应 1080p 远距目标）
- **输出**: 人体 bounding box（类别仅 person）+ 置信度
- **模型大小目标**: INT8 权重 ≤ 2.5M
- **抽帧策略**: 每 2~3 帧跑一次（等效 8~12 fps），两帧之间由跟踪器预测补位

### 2.2 模块2：多目标跟踪
- **选型**: ByteTrack 的端侧裁剪版（无外观 ReID 网络）
- **核心算法**: 卡尔曼滤波预测 + 匈牙利算法匹配
- **优化点**:
  - 卡尔曼状态压缩至 4/6 维（位置+速度）
  - 控制低分检测框数量，减少匹配开销
  - `max_age` 和 `min_hits` 针对 8~12 fps 检测频率调参

### 2.3 模块3：17 关键点姿态估计
- **关键点定义**: COCO 格式 17 点（nose, eyes×2, ears×2, shoulders×2, elbows×2, wrists×2, hips×2, knees×2, ankles×2）
- **网络**: YOLOv8n-pose（全帧推理，IoU 匹配关联到各 track 的检测框）
- **输入尺寸**: 全帧输入，输出 greedily 按 IoU≥0.1 匹配到输入 bbox
- **模型大小目标**: INT8 权重 ≤ 2.5M
- **运行策略**: 每检测帧运行一次，为规则引擎和分类器同时提供输入；跳帧时复用缓存关键点
- **训练策略**: 先用 COCO keypoints 公开数据预训练 → 再在跌倒场景数据上做 fine-tune

### 2.4 模块4：分类器（每帧都推理）
分类器对每个活跃 track **每帧都运行**，输出跌倒概率 `S_cls`。

- **检测帧**：使用当前帧姿态估计得到的 `track_kpts` 运行分类器
- **跳帧**：检测器和姿态估计器不运行，分类器仍基于跟踪器预测的 bbox 和缓存/预测的关键点继续推理

**两种实现**：
1. **简单图像分类器** (`SimpleFallClassifier`)：单分支 ROI 96×96 → 2-class logits
2. **融合分类器** (`FallClassifier`)：ROI 图像 + 17 keypoints + 8-d 运动特征 → sigmoid

分类器得分 `S_cls` 会实时送入规则引擎辅助姿态分类，也直接参与融合决策的时序平滑。

### 2.5 模块5：规则引擎（含分类器辅助姿态修正）
规则引擎持续低算力运行，基于连续 N 帧的 17 关键点 + Track 历史轨迹 + `cls_score`，输出疑似跌倒得分 `S_rule`（六条规则等权平均）。

#### A. 高度压缩 + 多点贴地
- `H_ratio = (head_y - ankle_y) / bbox_h`
- 正常站立 H_ratio ≈ 0.85~0.95；跌倒时大幅压缩
- 贴地点数 `N_ground`: 统计关键点满足 `y ≥ ground_y - ε` 的数量
- **触发条件**: `H_ratio < 0.6` 且 `N_ground ≥ 2`，或姿态已被判为 `lying`（含光轴方向跌倒）

#### B. 地面区域判定
- 地面区域由预设的梯形/四边形 ROI 标定
- 取可见关键点中 y 坐标最低的若干点，判断其是否落在地面 ROI 内
- **触发条件**: 至少 1 个贴地关键点在地面 ROI 内

#### C. 由动到静 / 持续静止
- 运动度量：Track 质心在时间窗口内的位移标准差
- 由动到静：前段 `avg_motion > motion_thresh` 且后段 `avg_motion < static_thresh`
- 若当前姿态为 `lying`，持续静止也可触发

#### D. 垂直方向快速下降
- 计算最近窗口内质心 y 坐标的垂直速度 `vy`
- **触发条件**: `vy > fall_vy_thresh` 且姿态为压缩/lying

#### E. 加速度辅助
- 计算垂直方向加速度
- **触发条件**: 向下加速度 `> accel_thresh` 且姿态为压缩/lying

#### F. 关键点完全不可见
- **触发条件**: 可见比例 `< no_keypoint_thresh`（遮挡或极端姿态）

#### 姿态分类与分类器辅助
`_classify_posture` 综合 `h_ratio`、`kpt_aspect`（关键点水平/垂直跨度）、`torso_angle`（肩-髋与垂直方向夹角）以及 `cls_score`：
- `cls_score > cls_posture_t1` → 强制 `lying`
- `cls_score < cls_posture_t2` → 强制 `standing`
- `t2 ≤ cls_score ≤ t1` 且原姿态为 `standing` → 降级为 `sitting`
- 光轴方向跌倒检测：`kpt_aspect > 1.5` 或 `torso_angle > 55°` 直接判为 `lying`

```
S_rule = (I(A) + I(B) + I(C) + I(D) + I(E) + I(F)) / 6.0
```

### 2.6 模块6：融合决策器
```
S_final = α·S_rule + β·S_cls + γ·S_temporal
```
- `α = 0.35`, `β = 0.45`, `γ = 0.20`（可配置）
- `S_temporal`：过去 K 帧（如 5~10 帧）分类器得分的滑动平均，抑制单帧抖动

**状态机**:
- `NORMAL → SUSPECTED → FALLING → ALARM_SENT → RECOVERING → NORMAL`
- 进入 `FALLING` 时触发单次告警 `should_alarm()`
- 姿态序列检查：要求最近窗口内包含 upright（standing/sitting）→ fall（crouching/lying）的转换，防止全程 lying 误报
- `cls_bypass_thresh`：当 `S_cls ≥ cls_bypass_thresh` 时，可绕过姿态序列检查并减半 `alarm_min_frames`，加速高置信度跌倒确认

**报警条件**:
- `S_final ≥ T_alarm`（建议初始 0.50~0.70）
- 且连续满足 ≥ `N_alarm` 帧（建议 0.5s~1.0s）

**复位条件**:
- `reset_seconds` 内不再满足报警条件则进入恢复；`recovery_seconds` 后回到 NORMAL

## 3. 测试流程（Test Pipeline）

### 3.1 阶段1：模块级单元测试
- **检测器**：mAP@0.5、mAP@0.5:0.95、Recall（针对小目标/遮挡人体单独评估）
- **跟踪器**：IDF1、MOTA、ID Switch 次数（输入为检测器输出的伪标注序列）
- **关键点**：PCK@0.2、OKS-AP（COCO val 预训练验证；量化后评估定点/浮点 L2 误差）
- **分类器**：Accuracy、Precision、Recall、F1、混淆矩阵（重点关注弯腰/蹲下/坐下等难例）
- **规则引擎**：规则触发率、误触发率（在非跌倒动作上的触发比例），通过自动化单测覆盖逻辑分支

### 3.2 阶段2：集成测试
- 检测 + 跟踪：抽帧场景下的 ID 连贯性、遮挡恢复
- 检测 + 跟踪 + 关键点：关键点对检测框偏移、尺度变化的鲁棒性
- 规则 + 分类器：规则误触发时分类器能否正确纠正（如蹲下被规则误判）
- 时序稳定性：同一段视频的多次运行结果方差（蒙特卡洛稳定性）

### 3.3 阶段3：端到端 Inference Pipeline 测试
- **视频级测试**：给定 1080p 视频，对比 GT 标注计算事件级 TP/FP/FN、漏报率、误报率/小时
- **实时性能测试**：在 x86（GPU）上测 FPS、延迟 P50/P99；评估不同抽帧率下的端到端延迟
- **鲁棒性测试**：光照变化、遮挡物、不同服装颜色、侧卧/仰卧/趴下等姿态

### 3.4 阶段4：端侧模拟部署验证
- **模型转换链路**：PyTorch → ONNX → 海思 wk/NNIE 格式。逐层验证输出 cosine similarity > 0.99
- **量化验证**：INT8 量化前后的精度掉点评估（检测 mAP、关键点 PCK、分类器 accuracy）
- **ARM 功能验证**：在 ARM CPU 上通过 ONNX/NCNN 跑通功能逻辑
- **资源评估**：峰值 DDR、Flash 占用、每帧理论+实测延迟

## 4. 分阶段训练流程（Training Pipeline）

训练依赖一套私有跌倒数据集，建议最低规模为：
- **跌倒正样本**：≥ 200 段有效 clip（覆盖不同年龄、服装、光照、跌倒方向）
- **非跌倒负样本**：≥ 400 段 clip（含蹲下、弯腰、坐下、正常行走等难例）
- 若实际数据量不足，优先保证难例多样性，并用数据增强和难例挖掘弥补

### 阶段1：人体检测器（独立可并行）
1. COCO person 预训练 80 epochs
2. 私有跌倒数据集 fine-tune（人体框标注）
3. 数据增强以轻度为主（Mosaic/Mixup 可选但端侧建议保守）
4. 导出 ONNX，为量化做准备

### 阶段2：17 关键点姿态估计（独立可并行）
1. COCO keypoints（或 CrowdPose）预训练权重加载（源分辨率通常为 256×192）
2. MobileNetV2/ShuffleNet backbone + 轻量 head
3. **在部署目标分辨率 128×128 上做 fine-tune**（必要时可用 96×96）
4. 若有业务需求，后续补充跌倒场景标注做 domain adaptation

### 阶段3：跟踪器（调参无 NN 训练）
1. 在阶段1检测器输出的私有视频序列上运行 ByteTrack-lite
2. 调优 `det_thresh`、`track_thresh`、`match_thresh`、`max_age`、`min_hits`
3. 评估 MOTA/IDF1，确保 ID Switch 可控

### 阶段4：分类器训练（串行，依赖前序模块）
1. 用阶段1检测器 + 阶段2关键点在训练视频上提取中间特征（ROI 图 + 关键点 + 运动轨迹）
2. 选择分类器类型：
   - **简单图像分类器** (`SimpleFallClassifier`)：仅 ROI 图像输入，适合端侧快速部署
   - **融合分类器** (`FallClassifier`)：ROI + 关键点 + 运动特征，精度更高
3. 训练 100~200 epochs，难例挖掘与数据补充（基于混淆矩阵分析）
4. 中间特征自动缓存到 pickle/NPY，避免反复前向推理。缓存策略：仅对跌倒/非跌倒有效 clip 进行抽取并降采样到 **5 fps**；单条样本约 50~80KB（ROI 用 uint8 压缩存储），1000 条样本约 50~80GB。训练环境需预留足够 SSD 空间，超限时启用 LRU 清理或分 shard 存储

### 阶段5：端到端串联与整体微调
1. 冻结检测器、关键点、分类器权重，在私有验证集上跑整条 pipeline
2. 若存在级联误差（如检测框抖动导致关键点不稳定），可做 **corrective fine-tune**
3. 在验证集上做阈值网格搜索：
   - `rules.trigger_thresh`
   - `fusion.alarm_thresh`
   - `rules.cls_posture_t1` / `cls_posture_t2`
   - `fusion.cls_bypass_thresh`
4. 输出最终部署模型 + 配置文件

## 5. 资源预算与模型规格

| 模块 | 输入尺寸 | INT8 权重目标 | 备注 |
|------|----------|---------------|------|
| 检测器 | 832×448 | ≤ 3.0M | 每 2~3 帧跑一次 |
| 关键点 | 全帧 | ≤ 2.5M | 每检测帧跑一次 |
| 分类器 | ROI 96×96 (+ 可选关键点+运动) | ≤ 0.5M | 每检测帧对每个 track 运行 |
| 跟踪器 | - | 无 NN 权重 | 纯 CPU 后处理 |
| **合计** | - | **≤ 6.0M** | 满足 30M Flash 限制 |

峰值 DDR 通过以下方式控制：
- 检测器输入 buffer 复用
- 关键点输入尺寸做小（128×128 上限）
- 中间特征图 channel 数压缩

## 6. 数据格式与接口约定

- **人体框标注**: COCO JSON 格式
- **跌倒事件标签**: 自定义 JSON/CSV（视频文件名、起止时间戳、跌倒/非跌倒标签）
- **中间特征缓存**: `.npy` / `.pkl`（检测框、关键点坐标、可见性、运动特征向量）
- **可复现性**: 每阶段固定随机种子、保存配置 yaml、保留最佳 ckpt

## 7. 风险与应对

| 风险 | 应对措施 |
|------|----------|
| 17 关键点网络在 96×96 输入下精度不足 | YOLOv8n-pose 全帧推理，无需逐 crop 运行 |
| 对着摄像头倒下/背对倒下时 h_ratio 变化小 | 引入 `kpt_aspect` + `torso_angle` 光轴跌倒检测，并用 `cls_score` 辅助姿态分类 |
| 3516C NPU 对某些算子不支持 | 训练阶段就约束算子列表（避免复杂上采样/自定义激活） |
| 规则阈值在不同摄像头/场景差异大 | 规则权重和阈值做成配置文件，现场可快速调参 |
| 分类器对检测框抖动敏感 | 加入时序滑动平均 + ROI 对齐裁剪 stabilize 输入；跳帧复用缓存分数 |
| 私有数据无关键点标注 | COCO 预训练后直接迁移；后续精度不足再补充标注 |
